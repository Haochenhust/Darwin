# 协作方式
1. PRD 就是契约，不要边做边改

一开始我尝试过让 AI 边执行边修改，想着这样可以灵活调整。结果发现完全让 AI 自己控制节奏是行不通的——没有约束的执行会出现大量偏差，前后端逻辑对不上、代码和实际需求脱节。

后来我明白：信息必须在一开始对齐，PRD 就是契约。**现在的做法是花足够时间把 PRD 写清楚——不是简单描述“要做什么”，而是明确“做成什么样”。** PRD 一旦定下来，就是 AI 执行的依据，不轻易变动。

2. 验收标准要量化
需求文档不能只写需求，必须写验收标准。比如 VLM 任务，验收标准从“生成好看的 html 页面”改成“布局还原度>90%，颜色匹配度>85%，包含指定的 5 个组件”。目标清晰了，AI 执行才有方向，验收也有依据。

3. 大任务必须拆解
每个子任务控制在 30 分钟内可完成，成功率远高于让 AI 一次干完。类似安装大量依赖（如 pytorch）、build 大型 Docker 镜像这种，就不适合用这套系统。

# 与AI讨论宏大问题
在与AI讨论宏大的问题时，要先与AI同步好context，并达成共识。而不是给他一个目标，然后让AI一步到位的完美执行。
先当“苏格拉底”，后当“项目经理”：在动手前，先让 AI 扮演一个“苏格拉底”式的提问者，通过不断追问来帮助我们理清思路、识别盲区，与 AI 充分对齐任务共识。
拆解任务，逐步喂养：将一个复杂的大问题，拆解成若干个边界清晰、输入输出明确的局部小问题。让 AI 逐个攻破，并在每个环节进行快速验证。当局部反馈符合预期后，再进入下一个环节。
通过这种“谋定而后动，拆解以致远”的方式，才能将一个宏大而模糊的目标，转化为 AI 可理解、可执行、可验证的行动路径。
低效对话 ❌
你：帮我设计一个AI行业动态采集的定时任务。
AI：好的，可以使用 cron 配合 web_search 工具，配置如下... (直接给出一个标准但可能不完全符合你需求的方案)
高效对话 ✅
你：我需要设计一个AI行业动态采集的定时任务。现在请你扮演一个顶级的AI产品经理和资深SRE，通过苏格拉底式提问法，帮我澄清所有的需求、约束和风险。
AI：乐意至极。让我们从目标开始：
1. (目标与价值) 这个任务的核心目标是什么？是为了个人学习，团队分享，还是竞品监控？产出的价值衡量标准是什么？
2. (输入与信源) 我们从哪些渠道采集信息？是特定的几个网站，还是宽泛的关键词搜索？信源的可靠性如何？
3. (处理与输出) 采集到的信息需要如何处理？是生成摘要，翻译，还是情感分析？最终的交付物是什么格式？（日报、卡片、写入数据库？）
4. ...
通过这种方式，AI 变成了一面帮助我们看见思维全貌的“镜子”，它会引导我们把一个模糊的想法，打磨成一个边界清晰、指标明确、风险可控的可执行方案，且这个环节也是我们与AI达成任务清晰共识的过程，让其执行的细节与我们预期能够匹配。
可复用提问脚手架模板：
我有一个初步想法：[一句话描述你的目标]

现在，请你从一个 [角色A] 和一个 [角色B]的视角，通过苏格拉底式提问法，包括且不限于从以下几个方面对我进行提问，帮助我澄清需求、约束和风险。一次只提一个问题。

1. **目标与价值 (Why)**：这个任务的最终目的是什么？解决了谁的什么问题？如何衡量它的成功？
2. **输入与范围 (What-In)**：需要处理哪些输入数据/信息？它们的格式、来源和质量如何？任务的边界在哪里？
3. **输出与交付 (What-Out)**：期望的最终产出是什么？它的格式、交付方式和接收者是谁？
4. **核心流程 (How)**：完成这个任务的关键步骤有哪些？是否存在依赖关系或顺序要求？
5. **施工约束 (Constraints)**：在技术、时间、成本或安全方面，有哪些必须遵守的限制？
6. **验收标准 (Quality)**：什么样的结果算是“好”的结果？有没有可以量化的指标？
7. **风险与预案 (Risk)**：这个任务在执行过程中可能遇到哪些主要风险？如果失败，最坏的结果是什么？是否有兜底方案？
8. **替代方案 (Alternatives)**：除了我提出的想法，是否还有其他更简单、更高效或更稳妥的实现路径？
可将这个模板存为个人 Snippet 或 OpenClaw 的一个 Skill，每次启动复杂任务前，先调用它与 AI 完成“思想的对齐”，完成在思路、步骤、验收标准共识后再去一步步让它迭代执行。

# 自己经验总结

让AI根据项目向自己提问，每次提一个问题，不断的让这个项目更清晰；
