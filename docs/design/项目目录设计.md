# 项目目录设计

## 设计思路：四层架构

在动手画目录之前，先明确我们之前讨论收敛出的核心架构分层：
┌─────────────────────────────────────┐
│  Layer 1: Channels (消息通道层)      │  飞书WebSocket、未来Telegram/Slack等
├─────────────────────────────────────┤
│  Layer 2: Orchestrator (编排层)      │  消息路由、GroupQueue、Session管理、定时调度
├─────────────────────────────────────┤
│  Layer 3: Agent (智能体层)           │  Claude Agent SDK query()、MessageStream、Hooks
├─────────────────────────────────────┤
│  Layer 4: Tools / MCP (能力层)       │  飞书API、文件操作、自定义MCP Tools
└─────────────────────────────────────┘
   ↕ 横切关注点: Storage(SQLite) / Config / Logger / Types

Channel → Orchestrator → Agent → Tools 的分层非常干净，每层有明确的职责边界。特别是 "编排层只做管道，不做智能" 这个原则，说明你对 Agent 系统的本质理解到位——Agent 的"智能"应该完全交给 LLM，编排器只管流程控制。

### 关键约束回顾（驱动结构设计的决策）：
| 决策                      | 对结构的影响                                                                 |
|---------------------------|-----------------------------------------------------------------------------|
| 单进程，无 Docker         | 无 container/子项目，agent 逻辑直接在 src/ 内                                 |
| Skills-first 哲学         | 能力扩展通过 MCP Tools (src/tools/)，不在编排器推功能                        |
| TypeScript                | 需要 tsconfig.json，编译输出到 dist/                                         |
| 飞书 WebSocket 长连接     | Channel 层需要处理私聊/普通群/话题群三种场景                                 |
| 每个 Group 独立 Session + CLAUDE.md | groups/ 目录按 group 隔离                                             |
| launchd 保活              | 需要 launchd/plist 文件                                                     |
| Vibe coding (Claude Code/Cursor) | 需要 CLAUDE.md + .claude/ 配置                                       |

### 关键设计决策说明
1. 为什么 tools/ 按领域再分子目录（core /feishu/general）
```
tools/
├── core/        # 编排器能力：发消息、定时任务、群组管理
├── feishu/      # 飞书平台能力：文档、日历、知识库
└── general/     # 通用能力：文件、剪贴板、系统信息
```
这不只是代码组织 — 它映射到 MCP Server 的命名空间：
```
// 可以按子目录注册多个 MCP Server，也可以合并为一个
allowedTools: ['mcp__clawbot__send_message',       // core/'mcp__clawbot__schedule_task',      // core/'mcp__feishu__create_doc',          // feishu/'mcp__feishu__query_calendar',      // feishu/'mcp__general__read_file',          // general/]
```
你可以选择一个统一 MCP Server（mcp__clawbot__*），也可以按领域拆成多个。前者更简单，后者更灵活（可以按 Group 选择性开放某些能力）。短期推荐统一一个，中期按需拆分。

2. 为什么 channels/feishu/ 有 5 个文件而不是 1 个
根据你之前的痛点，飞书接入的复杂度集中在三个点：
- 话题群线程管理 → 单独抽成 thread-manager.ts，复杂度隔离
- 消息体解析（飞书消息结构嵌套很深） → message-handler.ts
- 卡片消息构建 → card-builder.ts
这样每个文件职责单一，调试时不用在一个 500 行的大文件里翻来翻去。用 Claude Code / Cursor 做 vibe coding 时，指向具体文件也更高效。

3. 为什么 router.ts 不做内容清洗，只做路由
职责边界原则：**编排层的 router.ts 只处理"消息去哪"，不处理"消息长什么样"**。
- `router.ts` 负责：`formatInbound()`（给入站消息加时间戳/发送者标记）、`findChannel()`（根据 chatJid 找 Channel 实例）
- `agent/output-parser.ts` 负责：所有出站内容清洗（去除 `<internal>` 块、分割超长消息、结构化输出解析）

如果在 `router.ts` 里同时处理出站内容清洗，一旦输出格式发生变化（比如新增 `<thinking>` 标签的处理），就需要在 Orchestrator 层修改逻辑，违反分层原则。所有"内容如何呈现"的决策应该离 Agent 层尽可能近。

4. 为什么 session-manager.ts 从编排器里单独拆出
你之前遇到的 compacting 上下文混淆本质是 Session 管理问题。Session 的生命周期逻辑足够复杂，值得独立一个文件：
- 创建新 Session vs resume 旧 Session 的决策
- resumeSessionAt 的精确定位
- 主动 Session 切割（turns 过多时创建新 session）
- Session ID 与 Group 的映射持久化
4. 为什么 storage/ 用 Repository 模式而不是单个 db.ts
原设计把所有 5 张表的 CRUD 集中在一个 `db.ts`，随着业务增长会迅速膨胀为 500+ 行的上帝文件，且 Claude Code / Cursor 在 vibe coding 时无法精确定位"这是 session 相关的查询"还是"这是 task 相关的查询"。

**拆分原则：**
- `db.ts`：只做连接管理、WAL 模式开启、`withTx()` 事务封装。不包含任何业务逻辑。
- `repositories/<name>-repo.ts`：每张表一个文件，只操作自己的表，对外暴露语义化函数（而非 raw SQL）。

**调用方式：**
```typescript
// 其他模块按需引入对应 repo，不直接引入 db.ts
import { saveSession, findSessionByGroup } from '@/storage/repositories/session-repo';
import { createTask, findDueTasks } from '@/storage/repositories/task-repo';

// 跨表事务：通过 db.ts 的 withTx() 包装
import { withTx } from '@/storage/db';
withTx(() => {
  insertMessage(msg);
  updateGroupLastSeen(msg.chatId);
});
```

新增一张表 = 新增一个 repo 文件，不动其他任何文件。

5. 为什么 Scheduler 归入 orchestrator/ 而不是独立顶层目录
Scheduler 的本质是**一个时间驱动的消息来源**，和 Channel（飞书 WebSocket）是同等地位：

| 消息来源 | 触发方 | 投递目标 |
|---------|-------|---------|
| `channels/feishu/` | 用户发消息 | `GroupQueue` |
| `orchestrator/scheduler/` | 系统时钟到期 | `GroupQueue` |

两者最终都通过 `GroupQueue` 把消息推给 Agent，区别只在于触发源不同。把 Scheduler 放在顶层独立目录会误导读者认为它是一个与编排层平级的系统，但实际上它**强依赖** `GroupQueue`，本质是编排层的内部组件。

执行路径对比：
```
用户消息：飞书 WebSocket → Channel → DB(messages) → Orchestrator polling → GroupQueue → Agent
定时消息：cron 到期    → Scheduler.task-runner → GroupQueue → Agent
```

Scheduler 不经过 Channel 层（不需要 `receive_id_type`、`root_id` 等 IM 概念），直接构造 `ScheduledMessage` 推入 `GroupQueue`，由编排器统一调度。

6. 为什么 groups/ 在项目根目录而不是 data/ 下
因为 groups/<folder>/CLAUDE.md 是你需要手动编辑的文件（定义每个 Group 的 Agent 行为），它不是纯运行时数据。放在根目录下方便你直接打开编辑器修改，也方便 Claude Code 的 settingSources: ['project'] 发现它（cwd 指向 groups/<folder>/）。

7. 文件数量与 NanoClaw / OpenClaw 的对比
暂时无法在飞书文档外展示此内容
比 NanoClaw 多出的 5-10 个文件来自：
- 飞书 Channel 的细分（5 个文件 vs NanoClaw WhatsApp 的 1 个）— 因为飞书场景确实更复杂
- session-manager.ts 和 prompt-builder.ts 的独立抽取 — NanoClaw 把这些逻辑混在 agent-runner 里
- tools/feishu/ 的 MCP Tools — NanoClaw 没有飞书集成
但去掉了NanoClaw 的～1700 行 Docker 相关代码（container-runner、credential-proxy、IPC、mount-security 等），整体更精简。
